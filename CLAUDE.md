# Contract-OS — 작업 지침

㈜세움디자인하우징 전산 계약서 웹앱. 프론트엔드는 `public/`(정적), 저장 API는
Netlify Functions + Supabase(운영) / `server.js` + SQLite(로컬 개발).

## 배포 워크플로 (고정 규칙 — 항상 이대로)

작업을 마치면 **항상 다음 순서로 진행**한다. 사용자가 매번 따로 요청하지 않아도 된다.

0. **작업 시작 전, 작업 브랜치를 최신 `main`에 맞춘다** (`git fetch origin main && git reset --hard origin/main`).
   PR을 **squash 머지**하면 `main`이 재작성되므로, 재사용하는 작업 브랜치는 다음 작업 전에 반드시 `main` 기준으로 재설정해야 머지 충돌이 안 난다.
1. 그 세션에 지정된 작업 브랜치에 커밋하고 `origin`에 푸시한다.
2. **`main`을 대상으로 새 PR(풀 리퀘스트)을 생성**한다.
   - PR 제목·본문은 한국어로, 변경 요약을 간결하게.
3. **생성한 PR을 `main`에 자동 머지**한다.
   - 바로 머지가 가능하면 즉시 머지(`merge_pull_request`)한다.
   - 브랜치 보호 등으로 즉시 머지가 막히면 자동 머지를 활성화(`enable_pr_auto_merge`)한다.
   - 머지 방식은 **squash**를 기본으로 한다.
4. 머지되면 Netlify가 `main`에서 자동 배포하므로, 사용자에게 "배포 반영까지 보통 1~2분"이라고 안내한다.

> 참고: 이 저장소의 Netlify 운영 배포 브랜치는 `main`이다. `main`에 머지 = 운영 배포.
> 자동 머지는 리뷰 단계를 건너뛴다는 점(소유자 단독 운영이라 의도된 선택)을 인지하고 진행한다.

## 코드 관례

- 외부 런타임 의존성 최소화(브라우저 내장 API·Node 내장 모듈 우선). 주석·네이밍은 한국어 기존 스타일을 따른다.
- 계약 본문 전체는 `data`(jsonb/JSON)에 저장하고, 목록/검색용 요약 컬럼만 별도 추출한다.
- 서명·신분증 등 첨부 이미지는 `data` 안에 data URL로 보관하며, 무결성 봉인 해시에서는 제외한다.
- 신분증 등 내부 보관 자료는 `no-print`로 인쇄에서 제외한다.
